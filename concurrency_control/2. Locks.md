
## Row-Level Locks

In addition to table-level locks, there are row-level locks, which are listed as below with the contexts in which they are used automatically by PostgreSQL. See Table 13.3 for a complete table of row-level lock conflicts. Note that a transaction can hold conflicting locks on the same row, even in different sub transactions; but other than that, two transactions can never hold conflicting locks on the same row. Row-level locks do not affect data querying; they block only _writers and lockers_ to the same row. Row-level locks are released at transaction end or during
save point rollback, just like table-level locks.

### Row-Level Lock Modes

#### FOR UPDATE

FOR UPDATE causes the rows retrieved by the SELECT statement to be locked as though for update. This prevents them from being locked, modified or deleted by other transactions until the current transaction ends. That is, other transactions that attempt UPDATE, DELETE, SELECT FOR UPDATE, SELECT FOR NO KEY

UPDATE, SELECT FOR SHARE or SELECT FOR KEY SHARE of these rows will be blocked until the
current transaction ends; conversely, SELECT FOR UPDATE will wait for a concurrent transaction that has run any of those commands on the same row, and will then lock and return the updated row (or no row, if the row was deleted). Within a REPEATABLE READ or SERIALIZABLE transaction, however, an error will be thrown if a row to be locked has changed since the transaction started. For further discussion see Section 13.4.

The FOR UPDATE lock mode is also acquired by any DELETE on a row, and also by an UPDATE that modifies the values of certain columns. Currently, the set of columns considered for the UPDATE case are those that have a unique index on them that can be used in a foreign key (so partial indexes and expressional indexes are not considered), but this may change in the future.

#### FOR NO KEY UPDATE

Behaves similarly to FOR UPDATE, except that the lock acquired is weaker: this lock will not block SELECT FOR KEY SHARE commands that attempt to acquire a lock on the same rows. This lock mode is also acquired by any UPDATE that does not acquire a FOR UPDATE lock.


#### FOR SHARE

Behaves similarly to FOR NO KEY UPDATE, except that it acquires a shared lock rather than exclusive lock on each retrieved row. A shared lock blocks other transactions from performing UPDATE, DELETE, SELECT FOR UPDATE or SELECT FOR NO KEY UPDATE on these rows, but it does not prevent them from performing SELECT FOR SHARE or SELECT FOR KEY SHARE.

#### FOR KEY SHARE

Behaves similarly to FOR SHARE, except that the lock is weaker: SELECT FOR UPDATE is blocked, but not SELECT FOR NO KEY UPDATE. A key-shared lock blocks other transactions from performing DELETE or any UPDATE that changes the key values, but not other UPDATE, and neither does it prevent SELECT FOR NO KEY UPDATE, SELECT FOR SHARE, or SELECT FOR KEY SHARE. PostgreSQL doesn't remember any information about modified rows in memory, so there is no limit on the number of rows locked at one time. However, locking a row might cause a disk write, e.g., SELECT FOR UPDATE modifies selected rows to mark them locked, and so will result in disk writes.

# Locks

There are generally two types of locks in RDBMS

* **Shared lock** — multiple transactions can hold it on the same row simultaneously
	* other transactions can still read the row
	* no transaction can modify the row or acquire an exclusive lock on it

* **Exclusive lock** — only one transaction can hold it at a time
	* no other transaction can modify or lock the row
	* plain `SELECT` (without a lock clause) still reads the row via MVCC without needing a lock


### The Four Lock Modes

PostgreSQL uses four distinct row-level lock modes, ranging from weakest to strongest. Having different strengths allows the database to maximize concurrency. 

| Mode                        | Description                                                                                                                                              | Example                                                  |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| `FOR KEY SHARE` *(weakest)* | User is relying on this row existing. Others can still update non-key columns, but no one can delete the row or change its primary key.                  | `INSERT INTO posts (user_id) VALUES (1)` → locks user #1 |
| `FOR SHARE`                 | User is going to read the row and needs the data to stay exactly as-is. No one should change it — but multiple transactions can share this lock at once. | `SELECT * FROM accounts WHERE id = 1 FOR SHARE`          |
| `FOR NO KEY UPDATE`         | User is changing this row's data but not its primary key. Other transactions doing FK checks against this row can still proceed.                         | `UPDATE users SET status = 'active' WHERE id = 1`        |
| `FOR UPDATE` *(strongest)*  | User is going to change the primary key or delete the row — threatening the very existence of the row. No other transaction can lock or touch it.        | `DELETE FROM users WHERE id = 1`                         |


Within a transaction, Postgres automatically applies the appropriate lock based on the statements, and so on.
==You never have to think about this for write statements==; 
Postgres always ==picks the weakest lock== that keeps things safe.

Where you do take control is on ==`SELECT` statements==. 
A plain `SELECT` acquires no row lock at all. 
But you can explicitly request one using `FOR UPDATE`, `FOR SHARE`, or `FOR KEY SHARE`

### 1. `FOR KEY SHARE` (The "Just Checking" Lock)

This lock is very relaxed. It only exists to protect the row's identity (its Primary Key) from disappearing.

- **What IS Allowed:**
    
    - Others can read the row (`SELECT`).
    - Others can acquire their own `FOR KEY SHARE` or `FOR SHARE` locks.
    - Others can **update non-key columns** (they can acquire a `FOR NO KEY UPDATE` lock).
        
- **What is NOT Allowed:**
    
    - Others **cannot delete** the row.
    - Others **cannot update the Primary Key** or unique constraints.
    - (They cannot acquire a `FOR UPDATE` lock).
        

### 2. `FOR SHARE` (The "Strict Reader" Lock)

This lock freezes the data state. It allows multiple people to read it safely, but stops all modifications.

- **What IS Allowed:**

    - Others can read the row (`SELECT`).        
    - Others can acquire their own `FOR KEY SHARE` or `FOR SHARE` locks (multiple people can lock it for reading at the same time).
        
- **What is NOT Allowed:**
    
    - Others **cannot update any column** in the row (key or non-key).
    - Others **cannot delete** the row.
    - (They cannot acquire `FOR NO KEY UPDATE` or `FOR UPDATE` locks).
        

### 3. `FOR NO KEY UPDATE` (The "Standard Edit" Lock)

This is what happens when you do a normal `UPDATE`. It claims the row for editing, but leaves the door cracked open for foreign key checks.

- **What IS Allowed:**
    
    - Others can read the row (`SELECT`).
    - Others **can verify the row exists** (they can acquire a `FOR KEY SHARE` lock, which is why your updates don't block foreign key validations).
        
- **What is NOT Allowed:**
    
    - Others **cannot update** the row at all.
    - Others **cannot delete** the row.
    - Others **cannot strictly lock it for reading** (they cannot acquire a `FOR SHARE` lock).
        

### 4. `FOR UPDATE` (The "Total Lockdown" Lock)

This is the most aggressive lock, used for deletions or primary key changes. It claims total ownership of the row.

- **What IS Allowed:**
    
    - Others can read the row (`SELECT`) — _Remember, standard reads are never blocked! They will just read the old version of the row until your transaction finishes._
        
- **What is NOT Allowed:**
    
    - **Literally everything else.** * Others cannot verify the key (`FOR KEY SHARE`).
    - Others cannot lock it for reading (`FOR SHARE`).
    - Others cannot update non-key columns (`FOR NO KEY UPDATE`).
    - Others cannot update keys or delete it (`FOR UPDATE`).
### 2. Lock Wait Configurations (Handling Conflicts)

When you request a lock (e.g., using `SELECT * FROM my_table WHERE id = 1 FOR UPDATE;`), and someone else already holds a conflicting lock, the default behavior is to **block and wait forever** until they finish.

If you don't want to wait, PostgreSQL provides two crucial configurations you can append to your query:

#### `NOWAIT`

If the database cannot acquire the lock immediately, it will throw an error and abort your transaction.

- **Syntax:** `SELECT * FROM table FOR UPDATE NOWAIT;`
    
- **Use case:** When your application needs to either get the lock instantly or fail fast so it can try doing something else.

#### `SKIP LOCKED`

If the database encounters a row that is already locked by someone else, it completely ignores that row and moves on to the next one that matches your `WHERE` clause.

- **Syntax:** `SELECT * FROM table FOR UPDATE SKIP LOCKED;`
    
- **Use case:** Building highly concurrent worker queues.

### How PostgreSQL Automatically Assigns Locks

When you send a standard SQL command to PostgreSQL, it parses the query, looks at the columns you are touching, and automatically grabs the lowest possible lock required to safely execute the command.

Here are the general conditions and rules it follows:

**1. The "Just Looking" Rule (`SELECT`)**

- **Command:** Standard `SELECT * FROM table;`
    
- **Automatic Lock:** **None.** 

* **Why:** Because of PostgreSQL's MVCC (Multi-Version Concurrency Control) and snapshots, normal reads don't lock rows at all. They just read the version of the row that was valid when the query started.
    

**2. The "Foreign Key Safety" Rule**

- **Command:** `INSERT` or `UPDATE` on a table with a foreign key.
    
- **Automatic Lock:** **`FOR KEY SHARE`** (on the parent table).
    
- **Why:** If you insert a new `post` for `user_id = 1`, Postgres quickly grabs a `FOR KEY SHARE` lock on user #1. It's Postgres's way of saying, "Hold on, don't delete this user for a split second while I attach this post to them."
    

**3. The "Standard Modification" Rule**

- **Command:** `UPDATE table SET non_key_column = 'value';`
    
- **Automatic Lock:** **`FOR NO KEY UPDATE`**
    
-  **Why:** Postgres analyzes your query and realizes you are only changing things like a name, status, or date. Because you aren't touching the Primary Key or Unique constraints, it uses this slightly weaker lock to allow other transactions to still run foreign key checks against the row.
    

**4. The "Identity Change or Destruction" Rule**

- **Command:** `DELETE FROM table;` or `UPDATE table SET primary_key = 'new_value';`
    
- **Automatic Lock:** **`FOR UPDATE`**
    
- **Why:** You are fundamentally destroying the row or changing its core identity. Postgres automatically grabs the absolute heaviest lock, bringing a halt to anyone else trying to do _anything_ with that row.
