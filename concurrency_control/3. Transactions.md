# Isolation Levels

 **Read Committed Isolation Level**

_Read Committed_ is the default isolation level in PostgreSQL. When a transaction uses this isolation level, a

==SELECT query (without a FOR UPDATE/SHARE clause) sees only data committed before the query began==; it never sees either ==uncommitted data== or ==changes committed by concurrent transactions during the query's execution==. In effect, a SELECT query sees a snapshot of the database as of the instant the query begins to run. However,
SELECT does see the effects of previous updates executed within its own transaction, even though they are not yet committed. Also note that ==two successive SELECT commands can see different data==, even though they are within a single transaction, if other transactions commit changes after the first SELECT starts and before the second SELECT starts.

UPDATE, DELETE, SELECT FOR UPDATE, and SELECT FOR SHARE commands behave the same as SELECT in terms of searching for target rows: they will only find target rows that were committed as of the command start time. However, such a target row might have already been updated (or deleted or locked) by another concurrent transaction by the time it is found. In this case, ==the would-be updater will wait for the first updating transaction to commit or roll back== (if it is still in progress). ==If the first updater rolls back, then its effects are negated== and the second updater can proceed with updating the originally found row. If the first updater commits, the second updater will ignore the row if the first updater deleted it, otherwise it will attempt to apply its operation to the
updated version of the row. ==The search condition of the command (the WHERE clause) is re-evaluated to see if the updated version of the row still matches the search condition==. If so, the second updater proceeds with its operation using the updated version of the row. In the case of SELECT FOR UPDATE and SELECT FOR SHARE, this means it is the updated version of the row that is locked and returned to the client.


**Repeatable Read Isolation Level**

The _Repeatable Read_ isolation level only sees data committed before the transaction began; it never sees either uncommitted data or changes committed by concurrent transactions during the transaction's execution. (However, each query does see the effects of previous updates executed within its own transaction, even though they are not yet committed.) This is a stronger guarantee than is required by the SQL standard for this isolation level, and prevents all of the phenomena described in Table 13.1 except for serialization anomalies. As mentioned above, this is specifically allowed by the standard, which only describes the _minimum_ protections each isolation level must provide.

This level is different from Read Committed in that a query in a repeatable read transaction sees a snapshot as of the start of the first non-transaction-control statement in the _transaction_, not as of the start of the current statement within the transaction. Thus, successive SELECT commands within a _single_ transaction see the same data, i.e.,
they do not see changes made by other transactions that committed after their own transaction started.

Applications using this level must be prepared to retry transactions due to serialization failures.

UPDATE, DELETE, MERGE, SELECT FOR UPDATE, and SELECT FOR SHARE commands behave the same as SELECT in terms of searching for target rows: they will only find target rows that were committed as of the transaction start time. However, such a target row might have already been updated (or deleted or locked) by another concurrent transaction by the time it is found. In this case, the repeatable read transaction will wait for the first updating transaction to commit or roll back (if it is still in progress). If the first updater rolls back, then its effects are negated and the repeatable read transaction can proceed with updating the originally found row. ==But if the first updater commits (and actually updated or deleted the row, not just locked it) then the repeatable read
transaction will be rolled back with the message==

`ERROR: could not serialize access due to concurrent update`

because a repeatable read transaction cannot modify or lock rows changed by other transactions after the repeatable read transaction began.

When an application receives this error message, it should abort the current transaction and retry the whole transaction from the beginning. The second time through, the transaction will see the previously-committed change as part of its ==initial view of the database, so there is no logical conflict in using the new version of the row as the starting point for the new transaction's update.==

Note that only updating transactions might need to be retried; read-only transactions will never have serialization conflicts.

The Repeatable Read mode provides a rigorous guarantee that each transaction sees a completely stable view of the database. However, this view will not necessarily always be consistent with some serial (one at a time) execution of concurrent transactions of the same level. For example, even a read-only transaction at this level may see a control record updated to show that a batch has been completed but _not_ see one of the detail records which is logically part of the batch because it read an earlier revision of the control record. Attempts to enforce business
rules by transactions running at this isolation level are not likely to work correctly without careful use of explicit locks to block conflicting transactions.

## Isolation Levels at a Glance

| Anomaly               | Read Committed | Repeatable Read | Serializable |
|-----------------------|:--------------:|:---------------:|:------------:|
| Dirty Read            | ✓ prevented    | ✓ prevented     | ✓ prevented  |
| Non-repeatable Read   | ✗ possible     | ✓ prevented     | ✓ prevented  |
| Phantom Read          | ✗ possible     | ✓ prevented     | ✓ prevented  |
| Serialization Anomaly | ✗ possible     | ✗ possible      | ✓ prevented  |

> Postgres's default is **Read Committed**.

## Automatic Locks for Write Statements

Postgres automatically picks the **weakest lock that is safe** for each statement.
You never have to request these manually for writes — only for `SELECT`.

| Statement                       | Lock Auto-Applied |
| ------------------------------- | ----------------- |
| `SELECT`                        | None              |
| `SELECT FOR KEY SHARE`          | FOR KEY SHARE     |
| `SELECT FOR SHARE`              | FOR SHARE         |
| `SELECT FOR UPDATE`             | FOR UPDATE        |
| `UPDATE` on non-key columns     | FOR NO KEY UPDATE |
| `UPDATE` on primary key columns | FOR UPDATE        |
| `DELETE`                        | FOR UPDATE        |
| `INSERT` (FK check on parent)   | FOR KEY SHARE     |

## SELECT Behavior

A plain `SELECT` takes no row lock. ==It never waits== — it reads the committed version and moves on.

- Never sees uncommitted data
- Always sees its own work
- Can produce different values when re-run within the same transaction?
  - Yes — Read Committed (non-repeatable reads are possible; each statement gets a fresh snapshot)
  - No — Repeatable Read and Serializable (snapshot is fixed at transaction start)
- Sees committed data as of...
  - Start of the query — Read Committed
  - Start of the transaction — Repeatable Read, Serializable

## UPDATE / DELETE / SELECT FOR UPDATE Behavior

### Overview

`UPDATE`, `DELETE`, and `SELECT FOR UPDATE` are fundamentally different from a
plain `SELECT` in one critical way: they must **acquire a row-level exclusive
lock** on every row they intend to modify or hold. A plain `SELECT` reads
committed data through MVCC without acquiring any lock and without ever blocking.
Write statements cannot do this — two transactions cannot safely modify the same
row simultaneously, so they must coordinate through locking.

This introduces a class of problems plain `SELECT` never encounters: what happens
when the row you want to lock is already locked by another in-progress transaction?

---

### Phase 1 — Candidate Row Identification via Snapshot

When a write statement executes, Postgres first uses an MVCC snapshot to identify
**candidate rows** — the set of rows that satisfy the `WHERE` clause and are
eligible to be locked and operated on.

The point at which this snapshot is taken differs by isolation level:

- **Read Committed** — a fresh snapshot is taken at the start of each individual
  statement. If the transaction runs a second write statement later, it gets a
  new snapshot that reflects all commits that occurred since the first statement ran.

- **Repeatable Read / Serializable** — the snapshot is taken once at the start of
  the transaction and is reused for every statement within it. No commits from
  concurrent transactions become visible mid-transaction, regardless of how many
  statements execute.

This snapshot is used exclusively to identify candidates. It is not necessarily
the version of the row that ultimately gets locked or modified — that is
determined by the locking phase.

---

### Phase 2 — Lock Acquisition Per Row

For each candidate row, Postgres attempts to acquire a row-level lock. The lock
mode depends on the statement:

- `SELECT FOR UPDATE` → acquires `FOR UPDATE`
- `DELETE` → acquires `FOR UPDATE`
- `UPDATE` touching primary key or unique columns → acquires `FOR UPDATE`
- `UPDATE` touching only non-key columns → acquires `FOR NO KEY UPDATE`

If the row is **not currently locked**, the lock is granted immediately and the
operation proceeds.

If the row **is locked by another in-progress transaction**, Postgres cannot
proceed. It registers a lock dependency and enters a wait state, suspending
the current statement until the blocker commits or rolls back. Importantly,
only the specific locked row causes a wait — other candidate rows that are
free continue to be processed without delay.

---

### Phase 3 — Waiting and Deadlock Detection

While a transaction waits for a lock, Postgres runs its deadlock detector at
intervals defined by the `deadlock_timeout` setting (default: 1 second). The
detector inspects the **wait-for graph** — a directed graph where an edge from
transaction A to B means A is waiting for a lock currently held by B.

A **cycle** in this graph indicates a deadlock: a group of transactions are
each waiting on another in the group, so none can ever proceed on their own.
When a cycle is detected, Postgres selects one transaction as the victim and
aborts it:

```
ERROR:  deadlock detected
DETAIL: Process 42 waits for ShareLock on transaction 99;
        blocked by process 87.
HINT:   See server log for query details.
```

The victim is rolled back in full, all its locks are released, and the
remaining transactions in the cycle are unblocked. The victim is chosen based
on the estimated cost of rollback — typically the transaction that has done the
least work. If no cycle exists, the transaction continues waiting indefinitely
until the blocker finishes.

---

### Phase 4 — Post-Wait Re-evaluation

Once the blocking transaction finishes and the lock is released, Postgres does
**not** resume using the original snapshot version of the row. The row may have
changed. What happens next is determined by what the blocking transaction did
and which isolation level is in effect.

#### Blocking transaction committed a data change

The row now has a new committed version that differs from what the snapshot showed.

**Under Read Committed:**
Postgres re-fetches the latest committed version of the row and re-evaluates
the `WHERE` clause against it — a process referred to internally as
`EvalPlanQual`. Two outcomes are possible:

- The new version still satisfies the `WHERE` clause → the lock is acquired on
  the new version and the operation is applied to it. Note that the values
  being written are computed against this new version, not the snapshot version.
- The new version no longer satisfies the `WHERE` clause → the row is silently
  skipped and the statement moves on.

This means the actual set of rows affected by the statement can differ from the
candidate set the snapshot identified. A row that matched at statement start may
not match post-wait. The inverse is not possible — rows outside the original
candidate set are never picked up mid-statement.

**Under Repeatable Read / Serializable:**
Postgres detects that the row's current committed version was written by a
transaction that committed after your transaction's snapshot was taken. Allowing
the operation to proceed would mean acting on data your transaction was never
supposed to see, violating the isolation guarantee. Postgres aborts with:

```
ERROR:  could not serialize access due to concurrent update
```

The entire transaction is rolled back. The application is responsible for
retrying the transaction from scratch.

#### Blocking transaction deleted the row

The row no longer exists in any live heap version. There is nothing to lock
or operate on. Postgres skips the row silently and continues with the next
candidate. This behavior is identical across all isolation levels.

#### Blocking transaction rolled back

All changes made by the blocking transaction are discarded. The row reverts to
the version it held before that transaction began — which is the same version
your snapshot originally saw when it identified this row as a candidate.
Postgres re-evaluates the `WHERE` clause against this reverted version.

Since the row was already a candidate under the snapshot, it almost certainly
still satisfies the predicate. The lock is granted and the operation is applied.
This behavior is consistent across all isolation levels.

### Isolation Level Summary

| Behavior                                  | Read Committed     | Repeatable Read    | Serializable       |
| ----------------------------------------- | ------------------ | ------------------ | ------------------ |
| Snapshot taken at                         | Statement start    | Txn start          | Txn start          |
| Waits on locked rows                      | ✓                  | ✓                  | ✓                  |
| After wait: row modified by other txn     | Re-evaluates WHERE | Error — must retry | Error — must retry |
| After wait: row deleted by other txn      | Skips the row      | Skips the row      | Skips the row      |
| After wait: other txn rolled back         | Re-evaluates WHERE | Re-evaluates WHERE | Re-evaluates WHERE |
| Detects serialization anomalies           | ✗                  | ✗                  | ✓                  |

### How Postgres Processes Each Row

```
Scan rows using snapshot -> find candidates matching WHERE
      |
      +-> Row is FREE    -> lock it, apply change, done
      |
      +-> Row is LOCKED  -> check for deadlock
                                |
                          +-----+-----+
                          |           |
                       Deadlock    No Deadlock
                          |           |
                          v           v
                     Kill current   WAIT...
                     txn with error    |
                                  other txn finishes
                                       |
                            +----------+----------+
                            |          |          |
                         Updates    Deletes   Rollbacks
                            |          |          |
          RC:          Re-evaluate   Skip    Re-evaluate
          RR/Serial:     Error       Skip    Re-evaluate
```

### Deadlock

```
-- t(id, val): row 1, row 2

Time | Txn A                    | Txn B
-----|--------------------------|---------------------------
  1  | BEGIN;                   | BEGIN;
  2  | UPDATE t WHERE id = 1;   | UPDATE t WHERE id = 2;
     | -- locks row 1           | -- locks row 2
  3  | UPDATE t WHERE id = 2;   |
     | -- needs row 2 -> WAIT   |
  4  |                          | UPDATE t WHERE id = 1;
     |                          | -- needs row 1 -> WAIT
  5  | ERROR: deadlock detected |
     | -- Txn A rolled back     | -- gets lock, continues
```

Same behavior across all isolation levels.

### Lock Wait — Multiple Locked Rows

```
-- t(id, val): row 1=500, row 2=200, row 3=300, row 4=80
-- Txn B: UPDATE t SET val = val - 10 WHERE val > 100

Time | Txn A                    | Txn B
-----|--------------------------|---------------------------
  1  | BEGIN;                   | BEGIN;
  2  | UPDATE t SET val = 10    |
     |   WHERE id = 1;          |
     | -- row 1 locked (val=10) |
     | UPDATE t SET val = 250   |
     |   WHERE id = 2;          |
     | -- row 2 locked (val=250)|
  3  |                          | UPDATE t
     |                          |   SET val = val - 10
     |                          |   WHERE val > 100;
     |                          |
     |                          | -- id=1(500): locked -> WAIT
     |                          | -- id=2(200): locked -> WAIT
     |                          | -- id=3(300): free -> val=290
     |                          | -- id=4(80):  no match, skip
  4  | COMMIT;                  |
  5  |                          | -- wakes up on id=1 and id=2
     |                          |
     |                          | Read Committed:
     |                          |   id=1 val=10, 10>100? NO -> skip
     |                          |   id=2 val=250, 250>100? YES -> 240
     |                          |   Final: id=2(240), id=3(290)
     |                          |
     |                          | Repeatable Read / Serializable:
     |                          |   row was modified by committed txn
     |                          |   ERROR: could not serialize access
     |                          |   -- must retry whole transaction
```

### SELECT FOR UPDATE

```
-- t(id, val): row 1 val=500
-- Goal: deduct 200 only if val >= 200

Time | Txn A                    | Txn B
-----|--------------------------|---------------------------
  1  | BEGIN;                   | BEGIN;
  2  | SELECT * FROM t          |
     |   WHERE id = 1           |
     |   FOR UPDATE;            |
     | -- row 1 locked, val=500 |
  3  |                          | SELECT * FROM t
     |                          |   WHERE id = 1
     |                          |   FOR UPDATE;
     |                          | -- locked by Txn A -> WAIT
  4  | UPDATE t SET val = 300   |
     |   WHERE id = 1;          |
     | COMMIT;                  |
  5  |                          | Read Committed:
     |                          |   wakes up, reads val=300
     |                          |   300 >= 200 -> deduct -> val=100
     |                          |
     |                          | Repeatable Read / Serializable:
     |                          |   row modified by committed txn
     |                          |   ERROR: could not serialize access
```

Without `FOR UPDATE` both would read `500` at the same time, both approve,
and both deduct — final val would be `300` instead of `100`.

### SKIP LOCKED

```
-- jobs(id, status): id=1 pending, id=2 pending, id=3 pending

Time | Worker 1                 | Worker 2
-----|--------------------------|---------------------------
  1  | BEGIN;                   | BEGIN;
  2  | SELECT * FROM jobs       |
     |   WHERE status='pending' |
     |   LIMIT 1                |
     |   FOR UPDATE SKIP LOCKED;|
     | -- grabs id=1            |
  3  |                          | SELECT * FROM jobs
     |                          |   WHERE status='pending'
     |                          |   LIMIT 1
     |                          |   FOR UPDATE SKIP LOCKED;
     |                          | -- id=1 locked -> skip
     |                          | -- grabs id=2
  4  | COMMIT;                  | COMMIT;
```

Same behavior across all isolation levels.
