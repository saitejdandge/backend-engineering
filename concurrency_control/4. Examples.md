# Concurrency Control — Real World Examples

---

## 1. Inventory Management — Preventing Oversell

**Problem:** Two customers try to buy the last unit of a product simultaneously.
Without proper locking both reads see stock = 1, both approve the purchase, and
stock goes to -1.

**Isolation level:** Read Committed
**Lock used:** `SELECT FOR UPDATE`

```sql
BEGIN;

SELECT stock FROM inventory
  WHERE product_id = 42
  FOR UPDATE;           -- locks this row exclusively

-- application checks: stock >= quantity_requested
-- if yes:

UPDATE inventory
  SET stock = stock - 1
  WHERE product_id = 42;

COMMIT;
```

```
-- inventory: product_id=42, stock=1

Time | Txn A (Customer 1)            | Txn B (Customer 2)
-----|-------------------------------|-------------------------------
  1  | BEGIN;                        | BEGIN;
  2  | SELECT stock FROM inventory   |
     |   WHERE product_id = 42       |
     |   FOR UPDATE;                 |
     | -- stock=1, row locked        |
  3  |                               | SELECT stock FROM inventory
     |                               |   WHERE product_id = 42
     |                               |   FOR UPDATE;
     |                               | -- locked by Txn A -> WAIT
  4  | -- 1 >= 1, proceed            |
     | UPDATE inventory              |
     |   SET stock = stock - 1       |
     |   WHERE product_id = 42;      |
     | COMMIT; -- stock is now 0     |
  5  |                               | -- wakes up, reads stock = 0
     |                               | -- 0 >= 1? NO -> reject purchase
     |                               | ROLLBACK;
```

Txn B correctly rejects instead of creating negative stock.

## 2. Job Queue — Worker Picking

**Problem:** Multiple workers compete to claim pending jobs. Without the right
locking, workers either block each other or process the same job twice.

**Isolation level:** Read Committed
**Lock used:** `SELECT FOR UPDATE SKIP LOCKED`

```sql
BEGIN;

SELECT id, payload FROM jobs
  WHERE status = 'pending'
  ORDER BY created_at
  LIMIT 1
  FOR UPDATE SKIP LOCKED;  -- skip any job already claimed by another worker

-- if a row is returned:
UPDATE jobs
  SET status = 'processing', worker_id = $worker_id
  WHERE id = $id;

COMMIT;
```

```
-- jobs: id=1(pending), id=2(pending), id=3(pending)

Time | Worker 1             | Worker 2             | Worker 3
-----|----------------------|----------------------|---------------------
  1  | BEGIN;               | BEGIN;               | BEGIN;
  2  | SELECT...SKIP LOCKED |                      |
     | LIMIT 1;             |                      |
     | -- grabs id=1        |                      |
  3  |                      | SELECT...SKIP LOCKED |
     |                      | LIMIT 1;             |
     |                      | -- id=1 skip         |
     |                      | -- grabs id=2        |
  4  |                      |                      | SELECT...SKIP LOCKED
     |                      |                      | LIMIT 1;
     |                      |                      | -- id=1,2 skip
     |                      |                      | -- grabs id=3
  5  | -- process job 1     | -- process job 2     | -- process job 3
     | COMMIT;              | COMMIT;              | COMMIT;
```

All three workers run in parallel with zero contention. No job is processed twice.

## 3. Seat / Movie Booking — No Double Booking

**Problem:** Two users try to book the same seat. A plain read followed by
an insert is not safe — both can read the seat as available before either
completes the booking.

**Isolation level:** Read Committed
**Lock used:** `SELECT FOR UPDATE` on the seat row

```sql
BEGIN;

SELECT id, status FROM seats
  WHERE id = $seat_id AND status = 'available'
  FOR UPDATE;           -- lock the seat row

-- if row returned (seat is available):
INSERT INTO bookings (seat_id, user_id) VALUES ($seat_id, $user_id);

UPDATE seats SET status = 'booked' WHERE id = $seat_id;

COMMIT;
```

```
-- seats: id=5, status='available'

Time | Txn A (User 1)                | Txn B (User 2)
-----|-------------------------------|-------------------------------
  1  | BEGIN;                        | BEGIN;
  2  | SELECT id, status FROM seats  |
     |   WHERE id = 5                |
     |   AND status = 'available'    |
     |   FOR UPDATE;                 |
     | -- seat 5 locked, available   |
  3  |                               | SELECT id, status FROM seats
     |                               |   WHERE id = 5
     |                               |   AND status = 'available'
     |                               |   FOR UPDATE;
     |                               | -- locked by Txn A -> WAIT
  4  | INSERT INTO bookings ...      |
     | UPDATE seats SET              |
     |   status = 'booked'           |
     |   WHERE id = 5;               |
     | COMMIT;                       |
  5  |                               | -- wakes up, re-evaluates
     |                               | -- status = 'booked'
     |                               | -- 'booked' != 'available'
     |                               | -- WHERE fails -> no row returned
     |                               | -- seat unavailable, reject
     |                               | ROLLBACK;
```

The WHERE clause re-evaluation after the lock wait is what prevents the double
booking — Txn B's query returns no rows because the seat is now booked.

## 4. Bank Transfer — Preventing Serialization Anomaly

**Problem:** Two transactions each read one account's balance and update another's.
Under Read Committed or Repeatable Read they can interleave in a way that
violates business invariants — even without either transaction reading dirty data.

**Isolation level:** Serializable
**Why:** The anomaly is not a dirty read or a lost update — it is a
read-write dependency cycle that only Serializable can detect.

```
-- accounts: checking=500, savings=1000
-- Rule: total balance across both accounts must always be >= 1000

Time | Txn A                         | Txn B
-----|-------------------------------|-------------------------------
  1  | BEGIN ISOLATION LEVEL         | BEGIN ISOLATION LEVEL
     | SERIALIZABLE;                 | SERIALIZABLE;
  2  | SELECT balance FROM accounts  |
     |   WHERE type = 'savings';     |
     | -- sees 1000                  |
     | -- total is safe, deduct      |
     |   from checking               |
  3  |                               | SELECT balance FROM accounts
     |                               |   WHERE type = 'checking';
     |                               | -- sees 500
     |                               | -- total is safe, deduct
     |                               |   from savings
  4  | UPDATE accounts               |
     |   SET balance = balance - 600 |
     |   WHERE type = 'checking';    |
     | COMMIT; -- succeeds           |
  5  |                               | UPDATE accounts
     |                               |   SET balance = balance - 600
     |                               |   WHERE type = 'savings';
     |                               | COMMIT;
     |                               | ERROR: could not serialize access
     |                               | -- Txn B retries
```

Under Read Committed both commits succeed, leaving total balance at -100.
Under Serializable, Postgres detects the dependency cycle and aborts one.

## 5. Account Balance Transfer — Deadlock Avoidance

**Problem:** Transferring money between two accounts. If two transfers run
concurrently in opposite directions, they can deadlock.

**Isolation level:** Read Committed
**Lock used:** `SELECT FOR UPDATE` with consistent lock ordering

```sql
-- WRONG: locks accounts in arbitrary order -> can deadlock
BEGIN;
SELECT balance FROM accounts WHERE id = $from FOR UPDATE;
SELECT balance FROM accounts WHERE id = $to   FOR UPDATE;
...

-- RIGHT: always lock the lower id first
BEGIN;
SELECT balance FROM accounts
  WHERE id IN ($from, $to)
  ORDER BY id              -- consistent ordering prevents deadlock
  FOR UPDATE;
...
```

```
-- DEADLOCK SCENARIO (wrong ordering):
-- Transfer A->B and Transfer B->A running concurrently

Time | Txn A (A -> B)                | Txn B (B -> A)
-----|-------------------------------|-------------------------------
  1  | BEGIN;                        | BEGIN;
  2  | SELECT ... WHERE id=1         | SELECT ... WHERE id=2
     |   FOR UPDATE;                 |   FOR UPDATE;
     | -- locks account 1            | -- locks account 2
  3  | SELECT ... WHERE id=2         |
     |   FOR UPDATE;                 |
     | -- needs account 2 -> WAIT    |
  4  |                               | SELECT ... WHERE id=1
     |                               |   FOR UPDATE;
     |                               | -- needs account 1 -> WAIT
  5  | ERROR: deadlock detected      |
     | -- Txn A rolled back          | -- gets lock, continues

-- SAFE SCENARIO (consistent ordering):
Time | Txn A (A -> B)                | Txn B (B -> A)
-----|-------------------------------|-------------------------------
  1  | BEGIN;                        | BEGIN;
  2  | SELECT ... WHERE id IN (1,2)  | SELECT ... WHERE id IN (1,2)
     |   ORDER BY id FOR UPDATE;     |   ORDER BY id FOR UPDATE;
     | -- locks id=1 first           | -- tries id=1 first -> WAIT
  3  | -- then locks id=2            |
     | -- transfer, COMMIT           |
  4  |                               | -- wakes up, locks id=1, id=2
     |                               | -- transfer, COMMIT
```

## 6. Rate Limiting — Advisory Locks

**Problem:** You want to prevent more than one process from running a specific
operation (e.g. a scheduled report generation) at a time. There is no single
row to lock — the coordination is at the application level.

**Lock used:** PostgreSQL Advisory Locks
**Isolation level:** N/A — advisory locks are independent of transactions

Advisory locks are application-defined locks identified by a 64-bit integer
key (or two 32-bit integers). Postgres manages them but assigns no meaning to
them — the application decides what the key represents.

```sql
-- Session-level advisory lock (held until explicitly released or session ends)
SELECT pg_try_advisory_lock(12345);   -- returns true if acquired, false if not

-- Perform the protected operation only if lock was acquired
-- ...

SELECT pg_advisory_unlock(12345);

-- Transaction-level advisory lock (released automatically on commit/rollback)
SELECT pg_try_advisory_xact_lock(12345);
```

```
Time | Process 1                     | Process 2
-----|-------------------------------|-------------------------------
  1  | SELECT pg_try_advisory_lock   | SELECT pg_try_advisory_lock
     |   (12345);                    |   (12345);
     | -- returns true               | -- returns false (already held)
  2  | -- run report generation      | -- skip, another process
     |                               |   is already running
  3  | SELECT pg_advisory_unlock     |
     |   (12345);                    |
  4  |                               | -- can now acquire if needed
```

**Blocking variant** — waits instead of returning false:

```sql
SELECT pg_advisory_lock(12345);      -- blocks until lock is available
```

**Common use cases for advisory locks:**

- Ensuring only one instance of a cron job runs at a time
- Coordinating distributed processes that don't map to a single table row
- Application-level mutexes for operations that span multiple tables
- Locking a logical resource (e.g. "processing for tenant 42") without a dedicated lock row

## 7. Repeatable Read — Consistent Reporting

**Problem:** A report reads from multiple tables across several queries. Under
Read Committed, concurrent writes can cause the report to see inconsistent
data — some rows from before a commit and some from after.

**Isolation level:** Repeatable Read
**Lock used:** None — just the snapshot guarantee

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;

-- All queries in this transaction use the same snapshot.
-- Concurrent commits are invisible for the duration of the transaction.

SELECT SUM(amount) FROM orders   WHERE created_at >= '2024-01-01';
SELECT SUM(salary) FROM payroll  WHERE period = '2024-01';
SELECT COUNT(*)    FROM accounts WHERE status = 'active';

COMMIT;
```

```
Time | Report Txn (RR)               | Concurrent Write Txn
-----|-------------------------------|-------------------------------
  1  | BEGIN ISOLATION LEVEL         |
     | REPEATABLE READ;              |
     | -- snapshot fixed here        |
  2  | SELECT SUM(amount)            |
     |   FROM orders ...;            |
     | -- sees committed data        |
     |   as of step 1                |
  3  |                               | INSERT INTO orders ...;
     |                               | COMMIT;
  4  | SELECT SUM(salary)            |
     |   FROM payroll ...;           |
     | -- still sees snapshot        |
     |   from step 1                 |
     | -- new order is invisible     |
  5  | COMMIT;                       |
     | -- report is internally       |
     |   consistent                  |
```

Under Read Committed the report at step 4 would see the new order, creating
an inconsistency between the two query results within the same report.

## 8. FOR KEY SHARE — Comment Creation on a Post

**Problem:** A user creates a comment on a post. The comment insert references
the post via foreign key. You want to ensure the post is not deleted mid-insert,
but an admin should still be able to update the post title or content
concurrently — those changes do not threaten the comment's FK reference.

**Lock used:** `FOR KEY SHARE`
**Why not FOR UPDATE:** `FOR UPDATE` would block the title update too — more
than necessary. `FOR KEY SHARE` only prevents deletion and PK changes.

```sql
BEGIN;

SELECT id FROM posts
  WHERE id = $post_id
  FOR KEY SHARE;     -- ensures post cannot be deleted under us

INSERT INTO comments (post_id, body) VALUES ($post_id, $body);

COMMIT;
```

```
-- posts: id=10, title='Hello World'

Time | Comment Writer (Txn A)        | Admin (Txn B)
-----|-------------------------------|-------------------------------
  1  | BEGIN;                        | BEGIN;
  2  | SELECT id FROM posts          |
     |   WHERE id = 10               |
     |   FOR KEY SHARE;              |
     | -- KEY SHARE on post 10       |
  3  |                               | UPDATE posts
     |                               |   SET title = 'Updated'
     |                               |   WHERE id = 10;
     |                               | -- FOR NO KEY UPDATE:
     |                               | -- compatible with KEY SHARE
     |                               | -- proceeds immediately
  4  | INSERT INTO comments          |
     |   (post_id, body) VALUES      |
     |   (10, 'Great post!');        |
  5  |                               | DELETE FROM posts WHERE id=10;
     |                               | -- FOR UPDATE: conflicts with
     |                               | -- KEY SHARE -> WAIT
  6  | COMMIT;                       |
  7  |                               | -- lock released, delete proceeds
     |                               | COMMIT;
```

The title update at step 3 goes through immediately — `FOR NO KEY UPDATE` is
compatible with `FOR KEY SHARE`. The delete at step 5 waits because `FOR UPDATE`
conflicts with `FOR KEY SHARE`. Once the comment is safely inserted, the delete
is free to proceed.

## 9. FOR SHARE — Price Lock During Checkout

**Problem:** Multiple users check out the same product simultaneously. Each
checkout reads the price and records it on the order. A concurrent price update
by an admin should not silently change the price mid-checkout. Multiple checkouts
can proceed in parallel — only writes need to wait.

**Lock used:** `FOR SHARE`
**Why not FOR UPDATE:** `FOR UPDATE` would serialize all checkouts — only one
could hold the lock at a time. `FOR SHARE` lets multiple checkouts proceed in
parallel; only writes are blocked.

```sql
BEGIN;

SELECT id, price FROM products
  WHERE id = $product_id
  FOR SHARE;         -- price stays stable; concurrent checkouts can share lock

-- application verifies price matches what user was shown
INSERT INTO orders (product_id, price_at_purchase, user_id)
  VALUES ($product_id, $price, $user_id);

COMMIT;
```

```
-- products: id=7, price=99

Time | Checkout A           | Checkout B       | Admin
-----|----------------------|------------------|----------------------
  1  | BEGIN;               | BEGIN;           |
  2  | SELECT id, price     | SELECT id, price |
     |   FROM products      |   FROM products  |
     |   WHERE id=7         |   WHERE id=7     |
     |   FOR SHARE;         |   FOR SHARE;     |
     | -- price=99          | -- compatible    |
     | -- SHARED lock       | -- price=99      |
  3  |                      |                  | UPDATE products
     |                      |                  |   SET price=120
     |                      |                  |   WHERE id=7;
     |                      |                  | -- FOR NO KEY UPDATE
     |                      |                  | -- conflicts w/ SHARE
     |                      |                  | -- -> WAIT
  4  | INSERT INTO orders   | INSERT INTO      |
     |   price=99 ...;      |   orders         |
     | COMMIT;              |   price=99 ...;  |
     |                      | COMMIT;          |
  5  |                      |                  | -- released
     |                      |                  | -- price=120
     |                      |                  | COMMIT;
```

Both checkouts lock price at 99 and run in parallel with no contention between
them. The admin's price change waits until both commit. Future checkouts after
step 5 see price=120.

## Summary — Which Tool for Which Problem



| Scenario                        | Isolation Level  | Lock / Technique              |
|---------------------------------|------------------|-------------------------------|
| Inventory / ticket oversell     | Read Committed   | SELECT FOR UPDATE             |
| Job queue / worker pickup       | Read Committed   | SELECT FOR UPDATE SKIP LOCKED |
| Seat / resource booking         | Read Committed   | SELECT FOR UPDATE             |
| Cross-account transfer (safety) | Serializable     | Automatic                     |
| Deadlock avoidance              | Any              | Consistent lock ordering      |
| Distributed process mutex       | Any              | Advisory locks                |
| Consistent multi-query report   | Repeatable Read  | None (snapshot only)          |
| Protect parent during inserts   | Read Committed   | SELECT FOR KEY SHARE          |
| Consistent parent + child read  | Read Committed   | SELECT FOR SHARE              |
