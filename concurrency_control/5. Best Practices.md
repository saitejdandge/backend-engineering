# Concurrency Control — Best Practices

---

## 1. Use the Weakest Isolation Level That Guarantees Correctness

Higher isolation levels are not universally safer — they increase the chance
of serialization errors and transaction retries, which adds complexity to
application code and reduces throughput.

| If you need...                                          | Use...           |
|---------------------------------------------------------|------------------|
| Standard reads and writes with no special requirements  | Read Committed   |
| Consistent view across multiple queries in one txn      | Repeatable Read  |
| Protection against serialization anomalies              | Serializable     |

Default to Read Committed. Escalate only when you have a specific anomaly
to guard against that Read Committed cannot prevent.

---

## 2. Keep Transactions Short

Every row lock acquired inside a transaction is held until the transaction
commits or rolls back. Long-running transactions therefore hold locks for
longer, increasing the probability that other transactions will block.

**Avoid inside transactions:**
- Network calls (HTTP, gRPC) — variable latency, external failures
- Long-running business logic or heavy computation
- User interaction or waiting for input

**Pattern:** Do all preparation outside the transaction. Open the transaction,
run only the database statements, commit immediately.

```sql
-- BAD: network call inside transaction
BEGIN;
SELECT * FROM orders WHERE id = $id FOR UPDATE;
-- call external payment API...
UPDATE orders SET status = 'paid' WHERE id = $id;
COMMIT;

-- GOOD: prepare outside, transact fast
-- Step 1 (outside txn): call external payment API, get result
-- Step 2:
BEGIN;
SELECT * FROM orders WHERE id = $id FOR UPDATE;
-- verify state, apply result
UPDATE orders SET status = 'paid' WHERE id = $id;
COMMIT;
```

## 3. Lock Rows in a Consistent Order to Prevent Deadlocks

Deadlocks occur when two transactions each hold a lock the other needs.
The most reliable way to prevent them is to ensure every transaction that
touches multiple rows always acquires locks in the same order.

```sql
-- Consistent ordering by id prevents circular waits:
SELECT * FROM accounts
  WHERE id IN ($id1, $id2)
  ORDER BY id
  FOR UPDATE;
```

This is especially important in transfer-style operations where two
transactions commonly access the same pair of rows in opposite orders.

## 4. Always Retry on Serialization Errors

Under Repeatable Read and Serializable, concurrent updates to rows your
transaction has read will cause Postgres to abort your transaction with:

```
ERROR: could not serialize access due to concurrent update
```

Under Serializable, anti-dependency cycle detection can also abort:

```
ERROR: could not serialize access due to concurrent update
```

These are expected, not bugs. Applications using these isolation levels
**must** implement a retry loop around the transaction.

```python
MAX_RETRIES = 5
for attempt in range(MAX_RETRIES):
    try:
        with conn.transaction():
            # run the transaction
            break
    except SerializationFailure:
        if attempt == MAX_RETRIES - 1:
            raise
        time.sleep(backoff(attempt))
```

Read-only transactions under Repeatable Read will never encounter this
error — only transactions that write can be aborted this way.

## 5. Use SELECT FOR UPDATE for Read-Modify-Write Patterns

If your logic is: read a value, decide based on it, then write — you need
to lock the row during the read. Without it, another transaction can change
the row between your read and your write, making your decision stale.

```sql
-- UNSAFE: check then act without locking
BEGIN;
SELECT stock FROM inventory WHERE id = 42; -- sees stock=1
-- another txn also reads 1 here
UPDATE inventory SET stock = stock - 1 WHERE id = 42; -- both deduct!
COMMIT;

-- SAFE: lock the row during the read
BEGIN;
SELECT stock FROM inventory WHERE id = 42 FOR UPDATE;
-- only one txn proceeds, the other waits
UPDATE inventory SET stock = stock - 1 WHERE id = 42;
COMMIT;
```

This pattern applies to: inventory deduction, coupon redemption, seat
booking, any counter that must not go negative, and any conditional update
where the condition is checked in application code.

---

## 6. Use SKIP LOCKED for Job Queues and Work Distribution

`FOR UPDATE SKIP LOCKED` is the correct pattern for implementing work queues
in Postgres. It allows multiple workers to each claim a different row
atomically with zero blocking between workers.

```sql
BEGIN;
SELECT id, payload FROM jobs
  WHERE status = 'pending'
  ORDER BY created_at
  LIMIT 1
  FOR UPDATE SKIP LOCKED;
-- mark as in-progress, commit
COMMIT;
```

**Do not** use plain `FOR UPDATE` without `SKIP LOCKED` for queues — all
workers will serialize on the same row and throughput collapses to single-
threaded. **Do not** use `NOWAIT` for queues — it errors rather than moving
to the next available row.

---

## 7. Use NOWAIT for Fast-Fail Patterns

When your application cannot tolerate waiting for a lock — for example, a
UI operation that needs a result within a timeout budget, or a background
process where retrying is cheap — use `NOWAIT` to fail fast and handle the
contention explicitly.

```sql
BEGIN;
SELECT * FROM resource WHERE id = $id FOR UPDATE NOWAIT;
-- if lock is unavailable, Postgres raises an error immediately
-- application catches it and schedules a retry or returns "try again"
COMMIT;
```

This is preferable to a long lock wait when the downstream behavior on
conflict is well-defined and a wait would produce worse user experience.

---

## 8. Index Your WHERE Clauses — Lock Precision Matters

Postgres acquires row-level locks on individual rows, but it can only do
so efficiently if it can identify those rows efficiently. A table scan on
an unindexed predicate touches every row in the table, which means more
rows are considered as candidates and potentially more rows are locked.

Beyond performance, a full table scan under a concurrent workload increases
the chance that your query encounters locked rows it doesn't need, creating
unnecessary waits or deadlocks.

**Ensure there is an index** on every column used in the WHERE clause of
statements that acquire locks (`SELECT FOR UPDATE`, `UPDATE`, `DELETE`).

---

## 9. Prefer FOR NO KEY UPDATE Over FOR UPDATE When Possible

`FOR UPDATE` is the strongest lock mode and conflicts with `FOR KEY SHARE`.
This is a problem in schemas with foreign key relationships — an FK check
on a parent row takes `FOR KEY SHARE` on that row, and `FOR UPDATE` will
block it.

If your update does not touch primary key or unique constraint columns,
you do not need `FOR UPDATE`. Postgres will automatically use
`FOR NO KEY UPDATE` for such updates, which is compatible with `FOR KEY SHARE`.

Only use explicit `FOR UPDATE` (e.g. in a `SELECT FOR UPDATE`) when you
intend to subsequently delete the row or update its primary key.

---

## 10. Use Advisory Locks for Application-Level Coordination

When the resource you want to protect does not map neatly to a single row —
a global singleton operation, a per-tenant mutex, a distributed cron job —
use PostgreSQL advisory locks rather than inventing a lock table.

```sql
-- Acquire a lock keyed by an integer you define
-- (use hashing or a naming convention to assign integers to logical resources)
SELECT pg_try_advisory_xact_lock(hashtext('report:tenant:42'));
```

Advisory locks are cheap (held in shared memory), automatically released
on transaction end (with `_xact_` variants) or session end, and do not
require schema changes.

---

## 11. Do Not Hold Transactions Open Across User Interaction

Opening a transaction, showing a form to the user, and waiting for them to
submit before committing is a recipe for long-held locks. Users may close
tabs, go idle, or take minutes to respond. Any row locked by the idle
transaction blocks concurrent writes for that entire duration.

**Pattern:** Use optimistic concurrency control for user-facing flows.
Record a version number or timestamp when you show the data. When the user
submits, verify the version has not changed in the same transaction that
applies the update. If it has, show a conflict message and reload.

---

## 12. Monitor Lock Waits in Production

Lock contention that is acceptable in testing often becomes a bottleneck
under production load. Useful Postgres views for observing lock state:

```sql
-- Active lock waits right now
SELECT pid, wait_event_type, wait_event, query, state
  FROM pg_stat_activity
  WHERE wait_event_type = 'Lock';

-- Which process is blocking which
SELECT blocking_pid, blocked_pid, blocked_query
  FROM pg_stat_activity AS a
  JOIN pg_stat_activity AS b
    ON a.pid = ANY(pg_blocking_pids(b.pid))
  WHERE cardinality(pg_blocking_pids(b.pid)) > 0;

-- Lock details
SELECT pid, relation::regclass, mode, granted
  FROM pg_locks
  WHERE relation IS NOT NULL
  ORDER BY relation, pid;
```

High lock wait counts or long wait durations are symptoms of transactions
that are too long, lock ordering problems, or isolation level mismatches.

## Quick Reference

| Practice                              | Why                                          |
|---------------------------------------|----------------------------------------------|
| Weakest sufficient isolation level    | Less retry complexity, higher throughput     |
| Short transactions                    | Locks released faster, less blocking         |
| Consistent lock ordering              | Eliminates deadlocks                         |
| Retry on serialization error          | Required for RR / Serializable correctness   |
| SELECT FOR UPDATE before app-checked reads | Prevents lost updates                   |
| SKIP LOCKED for queues                | Zero-contention parallel workers             |
| NOWAIT for fast-fail paths            | Predictable behavior under contention        |
| Index WHERE clauses on write queries  | Lock fewer unnecessary rows                  |
| FOR NO KEY UPDATE where sufficient    | Allows concurrent FK checks                  |
| Advisory locks for logical resources  | No schema overhead for app-level mutexes     |
| No transactions across user input     | Prevents long-held locks from idle sessions  |
| Monitor pg_stat_activity / pg_locks   | Detect contention before it becomes an incident |
