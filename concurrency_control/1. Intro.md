# Concurrency control

These two concepts represent the fundamental philosophical divide in how databases handle multiple users trying to touch the same data at the same time.

If pessimistic concurrency is like securely booking a private meeting room, optimistic concurrency is like collaborating on a shared Google Doc and dealing with a merge conflict if two people edit the exact same sentence.

Here is a clear breakdown of the concepts, how traditional databases handled them, and exactly where PostgreSQL fits in.

### 1. The Concepts

#### Pessimistic Concurrency (Assume the Worst)

Pessimistic concurrency assumes that data conflicts are highly likely. Its primary rule is: **Lock it down before you touch it.**

- **How it works:** When a transaction wants to read or modify a row, it places a lock on it immediately. No one else can modify (and sometimes even read) that data until the first transaction is completely finished and releases the lock.
    
- **The Trade-off:** It guarantees absolute data safety and zero merge conflicts, but it kills performance. Transactions spend a lot of time waiting in line, and there is a high risk of "deadlocks" (where Transaction A is waiting for B, and B is waiting for A, so both freeze forever).
    
#### Optimistic Concurrency (Assume the Best)

Optimistic concurrency assumes that data conflicts are rare. Its primary rule is: **Do the work now, apologize (and retry) later if there's a problem.**

- **How it works:** Transactions don't lock the data. They read the data, make their changes locally, and right at the moment they attempt to `COMMIT`, the database checks to see if anyone else modified that exact same data in the background.
    
- **The Trade-off:** If there is no conflict, the database runs incredibly fast with zero waiting. If there _is_ a conflict, the database rejects the commit, throws an error, and forces the application to roll back and start the whole transaction over again.


### 2. How Traditional Databases Were Implemented

Historically, early relational databases (and early versions of systems like IBM DB2 or Microsoft SQL Server) relied heavily on **Strict Two-Phase Locking (2PL)**, which is a purely pessimistic approach.

In these traditional implementations:

- If you were writing to a row, nobody else could read it.
    
- If you were reading a row, nobody else could write to it.
    

This led to severe bottlenecks. A long-running analytics report (`SELECT`) could freeze an entire application because it was locking rows that the application was trying to update. To get around this, developers often had to write messy code or accept "dirty reads" (reading uncommitted, potentially invalid data) just to keep the system moving.

### 3. Does PostgreSQL Fall Under Both?

**Yes, absolutely.** PostgreSQL is a modern hybrid. It gives you the performance benefits of an optimistic system, but provides the tools to enforce strict pessimistic locks when you need them.

PostgreSQL achieves this using an architecture called **MVCC (Multi-Version Concurrency Control)**. Instead of overwriting data, MVCC creates _new versions_ of rows when they are updated. This means **readers never block writers, and writers never block readers.**

Here is how Postgres implements both sides of the coin:

#### PostgreSQL's Optimistic Side

- **Snapshot Isolation:** As we saw with the **Repeatable Read** isolation level, Postgres takes a snapshot of the data. If you try to commit an update, and Postgres realizes someone else changed the row after your snapshot started, it throws a `could not serialize access` error. This is textbook optimistic concurrency.
    
- **Application-Level OCC:** Developers often build explicit optimistic concurrency into Postgres applications by adding a `version` column to their tables.
    
    - `UPDATE users SET name = 'Bob', version = 2 WHERE id = 1 AND version = 1;`
        
    - If the row was updated by someone else, `version` is no longer `1`, the update affects 0 rows, and the app knows to retry.
        

#### PostgreSQL's Pessimistic Side

- **Default Write Behavior:** Even in the default `Read Committed` mode, when you issue an `UPDATE`, Postgres grabs a row-level lock (as we saw in the ASCII diagrams). If someone else is updating the row, you wait.
    
- **Explicit Locking:** Commands like `SELECT ... FOR UPDATE` are explicit pessimistic locks. You are telling Postgres, "I know a conflict is coming, so lock this row down right now before I even do my calculations."
