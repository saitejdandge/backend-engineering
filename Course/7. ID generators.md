# ID-Generator Problem statement

Assign a globally unique ID to `anything`
Write a function that spits out something  `unique` everytime it's invoked
*Note*: the function we are writing is a part of the application logic and NOT a central service.
## But why ?

Let's say we have sharded database, if we have  `auto increment` key this will not work because it's going to violate the `uniqueness` constraint. 

Therefore typically when the database is sharded, we need  `id generator logic` that can provide us unique id's across the shards when asked. we provide the `ids` during insert

## Time as ID

Time always move forward, Every time the function is invoked the time would have moved forward  & hence it seems unique !, *But what's the catch ?*

```
fun getId() {
	return get_epoch_ms()
}
```

The ID generation logic works fine when there is just one machine and the code is **not** invoked twice within one ms.

### Multiple machines:

Let's say we now have multiple machines calling the epoch time, if two machines call this function at exact same time, we would have a **collision**

![[Screenshot 2026-06-21 at 2.38.51 PM.png]]


#### What if we attach machine id as prefix

![[Screenshot 2026-06-21 at 2.39.20 PM.png]]

fun getId() {

### What if we have multiple threads within a machine

Now, even with machine id prefix, this would cause collision because we might have two different threads within a machine calling `getId` at exact time. We can have

`machineId + threadId + timestamp` as id.

```
fun getId(){
	return machineId + threadId + get_epoch_ms()
}
```

![[Screenshot 2026-06-21 at 2.42.31 PM.png]]

Another way to do is do with `atomic counter`

`id = machine_id + timestamp + counter++` 

*Note counter here is thread safe*
![[Screenshot 2026-06-21 at 2.44.00 PM.png]]
*Note: Now timestamp is not necessary and redundant, we can make id leaner by removing ts*

`id = machine_id + counter++`

![[Screenshot 2026-06-21 at 2.45.44 PM.png]]

```
int counter = 0
fun getId(){
	return machine_id + counter++
}
```
*Note if machine restarts counter resets to 0 (this is problem)*

#### We can save the counter to disk and resume post restart

```
int counter = load()
fun getId(){
	counter++        // atomic
	save_counter()   // DISK I/O
	return machine_id + counter
}
```

So when machine restarts, counter will not start from 0.

Problem: 
we are doing disk I/O every single time, we have to avoid this. Let's solve this by not writing to disk every single time, we can write in batches


```
int counter = load() + 1000 // flush frequency
fun getId() {
	counter++
	if(counter % 1000 == 0){
		save_counter()  // DISK I/O
	}
    return machine_id + counter
}
```

Framework of opposites: Time vs Frequency, Time not so reliable, because we don't know safest logic, due to correctness is important.

## When we have UUID's why are we generating ids ?

UUID's are 128 bit integers and are very in-efficient at scale

* They do not index well
* bloats up the index, index record will look like (index_id, row_id(primary key for eg))
* indicies are stored in memory and also persisted at disk. Typically good metric is RAM of instance should be at least cumulative size of the indicies.

They are random -> Good for security.

### 4 bytes vs 16 bytes integers

Your indices are `3x` times larger, your index lookups are disk IO. so, DB performance takes a hit.

## Cockroach DB vs Typical SQL database

Cockroach db recommends `UUID`, Typical RDBMS (B+ tree based) recommends `monotonically increasing`

In SQL database, we will insert at leaf nodes, if we use monotonically increasing value, these leaf nodes will be added to the right of the tree causing minimal re-balancing of tree.

Reason is ==cockroach db is distributed sql database==

cockroach db does range based partitioning, therefore if it's monotonically increasing causes hot shard problem. So that insertion goes to any node.

## MongoDB

MongoDB automatically assigns id, they are `12 bytes` long, they have 

`{timestamp-4b}{random-5b}{counter-3b}`

Mongo db also doesn't index well.

# Snow flake Twitter

*Adopted at discord & instagram*

Snowflakes are `64 bit` integers

`{timestamp-41b}{machineid-10b}{counter-12b(per machine)}`

This is not a service, this is a function. Snowflake is not a central service, instead it runs in app servers as a native function.

![[Screenshot 2026-06-21 at 6.16.00 PM.png]]

Every machine when boots up, it gets machine id.

Snowflakes are roughly sortable

* First 41 bits are epoch milli seconds, with every ms the ID will be moving forward.

This gives us an ability to get objects based on
* before/after a certain time
* before/after a certain id

Hence, pretty awesome for pagination, Twitter's pagination is not limit/offset based, 
instead it uses  `since_id` to paginated.

==sort ability matters if we want to do `pagination`==


```
select * from users where id>{last_id} limit 5
```

when we have complex queried pagination, we have to use default way.

But if we have id based pagination, we can leverage the last_id. (ex: mongoDB)

TWITTER + MONGODB + DYNAMO DB + ELASTIC SEARCH (scroll api) 
-> same patter id > {last_id}
