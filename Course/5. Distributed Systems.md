### Discussion over how LB server keeps track of all the application servers

![[Screenshot 2026-06-20 at 1.39.30 PM.png]]

How does `LBServer` know about `app servers`, we will need some layer of *persistence* at LB server

This is where we introduced configDB that saves key value pairs. 
example

```
lb1 -> [b1, b2,b3]
lb2 -> [b4,b5]
```

There are some problems with this approach, LB servers have to constantly lookup into config DB. This causes issues
* n/w latency
* db lookup, even 5ms for a millions of request will bring down the latency
* load on db increases, introduces complexity to scale up config db.
* cost increases due to infra reads from db.

We can solve them by introducing a cache layer at LB server. But now

**How does cache gets updated ?**

Let's assume we have an admin who adds up the new `app servers` through `LB console`

#### Option 1 (LB Console pushing updates to configDB + LB servers)

we  push the `update` to `config DB` & `LB servers`,
But we may have more than one LB server, how are we going to deal with it. This seems like `Fanning out` the update across multiple servers. How do we even know the nodes of `LB servers` in the first 

#### Option 2 (LB Servers polling updates from configDB)

We will have some interval of `30s`, every `30s` each LB server will look into `config DB` for update. But this doesn't scale when there is burst of traffic, `LB server` update will have to wait `30s` to increase the `app servers` causing full failure of systems


#### Option 3 (Message queue)

**we could introduce some sort of queue that would fanout to all the `LB servers` to update their cache**

Again, we have two options here, should we go with approach where
* Clients will pull the message (Kafka)
* Queue will push the message (Pub Sub Redis)

We can go with approach where  queue will push the message to  `LB servers`, because we almost want the update immediately and this is minor data change.

What if Queue goes down, LB servers won't get the updates to scale the `app servers`

>Always think about failure of component while designing high availability system
>Replication is fundamental thing to achieve availability

#### Option 4 (Message queue + LB servers polling from configDB) | Recommended

* Queue will be pushing the data change updates to `LB servers` to update their cache
* We will also ask `LB servers` to poll the data from `DBconfig` every `30s`

### Fault tolerance of configDB

To handle the failure of  `configDB`, we will introduce another `passive db

![[Screenshot 2026-06-20 at 2.00.58 PM.png]]

This entire box functionality can be replaced by **zoo keeper***

#### Zoo keeper

* Gives durable storage
* Gives config store
* Client can subscribe to key or path
* Any changes in the path will be propagated to client.
* Kind of `DynamoDB + Redis pub-sub`

### Tracking failed application nodes

#### Option 1 (App nodes telling, I'm alive...)

* Con - Application server have to implement the `/health` end point and constantly tell the `load balancer.
	* Poor user experience
#### Option 2 (LB server checks the health of app nodes, Are you alive ?..)

LB server periodically calls the health end points of the application server. 

* application server will have to implement health end point
* `config DB` will have another field call `health end point & metadata`
	* ```
	  b-servers: {l1:[b1,b2]..}
	  health_endpoint: {...}
	  ```

* Pro - Easy to implement
* Con - Application server have to implement the `/health` end point.
	* Poor user experience

#### What if there are multiple load balancer nodes

There could be multiple `lb nodes` those constantly make request to `app servers` increasing load on every `app server`

#### We will have separate component, orchestrator that does health checks

![[Screenshot 2026-06-20 at 2.27.58 PM.png]]

we will have standby mode to avoid  `orchestrator` failure to increase the `availability`

![[Screenshot 2026-06-20 at 2.29.36 PM.png]]

if health checks fail, we need to update the `cache` (through pub-sub send message) + update the db

Note: We will have shard instances of  `configDb`, `orchestrator + standy` for all the `lb nodes`


![[Screenshot 2026-06-20 at 2.35.33 PM.png]]

Orchestrator (one node) won't be enough to balance the `load balancer`.

> We have load balancer to balance the load to load balancer
> Whenever such recursion happens think of **LEADER ELECTION**

*Leader election* is end of recursion.

Instead of  `ACTIVE + PASSIVE` for `orchestrator`, we will have `ACTIVE + ACTIVE`
![[Screenshot 2026-06-20 at 2.38.47 PM.png]]

Worker does the heavy lifting, manager manages the workers.

*  Task distribution
	* `lb1, lb2.. -> w1`, `lb3, lb4.. -> w2``
	*  we will have persistence at `orchestrator` level.
- Monitor health of workers
	- Redistribute the tasks
* What if manager goes down, one of worker becomes the manager
	* Leader election

### Request propagation from client to LB servers

We will have multiple load balancer nodes, how does client know where to connect.

This can be done using **Connection balancers***

Clients talk to `load balancer` DNS, DNS will give the records of available `load balancer` i/p addresses

![[Screenshot 2026-06-20 at 2.54.19 PM.png]]

Clients will talk to domain, ex: `lb.google.com`

at DNS level we will have

```
lb.google.com -> 10.1.10
lb.google.com -> 10.2.10
```

### How do we scale the load balancer nodes

### Scaling DNS server

DNS should be highly  available.
We can't have `leader election` because workers will get different i/p address how do we talk to workers. 

We can have `stand by DNS`, very likely we would not another machine.

![[Screenshot 2026-06-20 at 3.01.38 PM.png]]

But who will monitor the health of this `Active DNS server` and change the i/p address at `source DNS`

Passive monitors Active through protocol - `VRRP Protocol`

if active is down, passive will advertise the virtual ip
![[Screenshot 2026-06-20 at 3.09.37 PM.png]]

![[Screenshot 2026-06-20 at 3.10.17 PM.png]]


Client knows virtual ip address of DNS server. virtual ip changes if there is change in active-passive nodes of `DNS server`
