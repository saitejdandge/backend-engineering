# Remote locks:

locks managed by central machine. -> *Lock Manager*

* Multiple threads synchronise through `Mutex & Semaphores`
* Multiple processes synchronise through `Disk`
	* ex: `apt-get upgrade` cannot be run concurrently
* Multiple machines synchronise through `Remote locks`

*Insight*: We sync using closest memory

*Conceptually we have a queue, where every consumer will have a lock using Redis lock manager, We want Atomic execution and Timeout TTL*


#### Consumer's pseudo code

```
ACQ_LOCK()
 
	READ_MESSAGE()

REL_LOCK()
```
*All consumers wait on ACQ_LOCK while one of them READ_MESSAGE()*


#### Requirements

- Atomic in nature
- Expiry

## Distributed locks (Redlock)

Idea: What we did in remote lock, just distribute it

#### Acquire lock

- Client goes through 5 nodes, trying to `ACQ_LOCK` with timeout
- If lock is acquired over majority > 50%, then it means we *Acquired*
- else the release the lock
